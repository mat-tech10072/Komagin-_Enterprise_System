# Komagin HR — Phase 5 Stage 5.9: Document QR Verification (Disabled)

**Document type:** Phase 5 Deliverable — Stage 5.9 Report
**Status:** Complete, live-verified.
**Date compiled:** 2026-07-13

---

## 1. Scope Decision (Recap)

KOM-097: the QR code option on generated documents linked to `/verify-doc.php?ref=<docnum>`, a public verification page that never existed anywhere in the codebase. The QR image itself was generated by an outbound call to a third-party API (`api.qrserver.com`) — an unauthorized external dependency this program's prohibited-actions list explicitly disallows. 0 of 47 live templates had the option enabled at the time of investigation (dormant, not actively broken for any real document today). Per user decision, the feature is **disabled** rather than completed — building a real public verification endpoint is a genuine feature requiring its own design/security decisions (public-facing, unauthenticated, needs its own scope), not something to build as a byproduct of closing out a dormant-config finding.

## 2. What Was Changed

- **`modules/documents/templates.php`**: removed the "QR Code" toggle from the template editor UI entirely — a user can no longer enable it going forward. The server-side `$showQR` value is now hardcoded to `0` (not merely omitted-from-form) regardless of what a request sends, so a crafted POST cannot re-enable it by adding the field back in directly.
- **`config/DocumentEngine.php`**: the QR-generation block (which built the `verify-doc.php` URL and the `api.qrserver.com` image tag) is removed outright — `$qrHtml` is now unconditionally an empty string, **regardless of what `show_qr_code` holds in the database**. This is deliberate defense-in-depth: even a template row with a stale/tampered `show_qr_code=1` value can no longer produce a QR image or call the external API. Also removed a vestigial `'qr_code'` entry from the template-variable HTML-escaping list, which had never actually corresponded to a real variable `buildVariables()`/`$extra` ever produced.
- **`show_qr_code` column**: left in place, untouched at the schema level — per the decision hierarchy's "don't remove data structures without migration/rollback planning," this is a UI/rendering-level deactivation, not a schema change. All 47 existing templates already had it at `0`; no data migration was needed.

## 3. Live Verification

1. **UI removal**: confirmed the "QR Code" toggle no longer appears anywhere in the template editor's rendered HTML.
2. **Server-side hardening**: submitted a crafted `POST` directly to `templates.php` with `show_qr_code=1` included (bypassing the UI entirely) — the saved row's `show_qr_code` was confirmed `0` in the database, proving the server does not trust the field even if a request supplies it.
3. **Defense-in-depth at generation time**: manually forced a disposable test template's `show_qr_code` to `1` directly in the database (simulating any stale/pre-existing row that might already have had it set), then generated a real document preview from that template — confirmed the output contains **zero** references to `qrserver`, `verify-doc`, or a QR `<img>` tag. This proves the rendering path itself no longer honors the flag at all, not just that new saves are blocked.
4. Disposable test template removed and verified absent afterward.
